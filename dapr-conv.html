Dapr의 Service Invocation 패턴을 TIBCO Rendezvous (RV) 기반 서비스들과 연동하려면, 단순히 HTTP/gRPC 호출을 “메시지 기반 미들웨어 호출”로 변환해주는 Gateway/Broker Layer를 설계하는 것이 핵심입니다.
TIBCO RV는 Publish/Subscribe 기반의 메시징 버스 구조를 가지고 있으므로, Dapr의 요청/응답 모델과 직접적으로 맞지 않습니다.
따라서 “Service별로 독립된 RV Daemon(rvd)”를 운영해야 하는 상황에서는 Dapr → RV Gateway → 각 RV 서비스 형태로 아키텍처를 분리하는 것이 가장 현실적이고 안정적입니다.

🔸 RV Gateway 설계 포인트

  1. 서비스별 연결 관리

    • Gateway 내부에서 서비스별로 rvd 인스턴스를 관리 (ex. Dictionary 형태)

    • 접속정보(daemon, network, service string 등)를 구성 파일이나 DB로 관리

    • 최초 호출 시 lazy-load 방식으로 해당 rvd 연결 생성

    csharp
    class RvConnectionManager {
    private readonly ConcurrentDictionary<string, RvConnection> _connections = new();

    public RvConnection GetOrCreate(string serviceName) {
        if (!_connections.TryGetValue(serviceName, out var conn)) {
            conn = new RvConnection(config[serviceName]);
            _connections[serviceName] = conn;
        }
        return conn;
    }
}

  2. 요청/응답 변환 Adapter

    • Dapr로 들어온 POST /invoke/rv-gateway/method/<svc> 요청을 받아
      해당 RV subject로 메시지를 publish하고 응답 메시지를 기다림

    • Request/Reply 패턴 구현 (inbox subject 사용)

    • RV 응답을 HTTP Response로 변환해 Dapr로 반환

  3. 구성 관리 (예: Dapr Secret Store + Config Store 활용)

    • 서비스별 network, service, daemon 정보를 Dapr Secret Store에 저장

    • 예:

      yaml
      kind: Secret
metadata:
  name: rv-conn-info
data:
  inventory:
    network: "10.0.0.1"
    service: "7600"
    daemon: "tcp:7600"

  4. 확장성

    • RV Gateway는 Dapr App으로서 Scale-out 가능

    • 각 인스턴스는 내부적으로 Connection Pool 또는 Session Cache 유지

    • Dapr의 Service Discovery (placement) 기능을 통해 로드 밸런싱 자동화

🔹 통신 흐름 예시

  1. inventory-api Dapr App이 rv-gateway 호출:

    bash
    POST /v1.0/invoke/rv-gateway/method/inventory
Body: { "productId": "A001" }

  2. rv-gateway는 내부적으로 TIBRV.SendRequest("inventory.request", body) 수행

  3. RV 응답 수신 시 변환 후 반환:

    json
    {
  "status": "OK",
  "price": 1200
}

🔸 확장 아키텍처: Multi-RV Gateway

서비스별 rvd 구성이 완전히 분리되어 있고, 서로 다른 보안/망 경계를 가진다면, 아래처럼 서비스 단위 Gateway 구성을 권장합니다.

css
[Dapr App] ─▶ [RV-GW-Inventory] ─▶ [rvd-inventory] ─▶ [RV Service: Inventory]
             ├▶ [RV-GW-Order] ─▶ [rvd-order] ─▶ [RV Service: Order]
             └▶ [RV-GW-Billing] ─▶ [rvd-billing] ─▶ [RV Service: Billing]

이 구조는:

  • 서비스별 인증/보안정책 격리

  • 장애 영향 최소화

  • Dapr에선 단순히 “service name”만 다르게 호출

✅ 정리

항목권장 접근
RV 접근방식Dapr Service Invocation → RV Gateway Adapter
rvd 관리서비스별 개별 인스턴스, Lazy init + Pooling
구성 관리Dapr Secret Store / Config Store 활용
장애 복구Gateway 레벨 Retry + Dapr Resiliency 정책
스케일링Gateway를 Dapr App으로 다중 배포 (service discovery 자동 처리)

⚙️ Services/RvConnectionManager.cs

csharp
using RvGateway.Models;
using TIBCO.Rendezvous;
using YamlDotNet.Serialization;
using YamlDotNet.Serialization.NamingConventions;
using System.Collections.Concurrent;

namespace RvGateway.Services;

public class RvConnectionManager
{
    private readonly string _configPath;
    private readonly ConcurrentDictionary<string, (TibrvTransport, RvConnectionConfig)> _connections = new();

    public RvConnectionManager(string configPath)
    {
        _configPath = configPath;
        Tibrv.Open(Tibrv.IMPL_NATIVE);
    }

    public RvConnectionConfig? GetConfig(string serviceName)
    {
        using var reader = new StreamReader(_configPath);
        var yaml = new DeserializerBuilder()
            .WithNamingConvention(CamelCaseNamingConvention.Instance)
            .Build();

        var configs = yaml.Deserialize<Dictionary<string, RvConnectionConfig>>(reader);
        return configs.TryGetValue(serviceName, out var cfg) ? cfg : null;
    }

    public TibrvTransport GetOrCreateTransport(string serviceName)
    {
        if (_connections.TryGetValue(serviceName, out var entry))
            return entry.Item1;

        var cfg = GetConfig(serviceName)
            ?? throw new Exception($"No config for service '{serviceName}'");

        var transport = new TibrvNetTransport(cfg.Service, cfg.Network, cfg.Daemon);
        _connections[serviceName] = (transport, cfg);
        return transport;
    }
}

csharp
using TIBCO.Rendezvous;
using RvGateway.Models;

namespace RvGateway.Services;

public class RvServiceInvoker
{
    private readonly RvConnectionManager _connMgr;

    public RvServiceInvoker(RvConnectionManager connMgr)
    {
        _connMgr = connMgr;
    }

    public string SendRequest(string serviceName, string requestJson)
    {
        var cfg = _connMgr.GetConfig(serviceName)
            ?? throw new Exception($"Service config not found: {serviceName}");

        var transport = _connMgr.GetOrCreateTransport(serviceName);
        using var msg = new TibrvMsg();
        msg.SendSubject = cfg.Subject;
        msg.AddField("data", requestJson);

        var listener = new TibrvTransport(transport);
        string replySubject = $"{cfg.ReplySubjectPrefix}.{Guid.NewGuid()}";

        // Send request and wait for reply
        var reply = TibrvMsg.SendRequest(msg, transport, 5000); // 5s timeout

        if (reply == null)
            throw new TimeoutException($"No reply from {cfg.Subject}");

        return reply.GetField("data")?.Value?.ToString() ?? "{}";
    }
}

1️⃣ Dapr 사이드카 통신 방식

항목설명
HTTP기본값. 클라이언트 SDK가 HTTP 요청을 만들어 자신의 사이드카로 전송 → Dapr는 서비스 디스커버리 후 대상 사이드카로 HTTP 호출
gRPCSDK 옵션에서 gRPC를 선택하면 클라이언트 사이드카와 서버 사이드카 간 gRPC 스트림으로 통신. REST보다 빠르고 바이너리 메시지 처리 가능

즉, Client App ↔ Client Sidecar는 SDK에서 선택,
Client Sidecar ↔ Server Sidecar는 Dapr 내부에서 자동 라우팅되며, 같은 프로토콜을 사용합니다.

2️⃣ 기본 흐름

arduino
Client App
   │ (Dapr SDK HTTP/gRPC)
   ▼
Client Sidecar
   │ (Dapr 내부 Service Invocation)
   ▼
Server Sidecar
   │ (HTTP/gRPC)
   ▼
rv-gateway App

  • 클라이언트 SDK가 HTTP 호출이면 사이드카가 내부적으로 HTTP 전송

  • SDK가 gRPC 호출이면 사이드카가 내부 gRPC 스트림 사용

  • 서버 사이드카는 애플리케이션이 어떤 포트/프로토콜을 쓰는지 맞춰 요청을 전달

기준HTTPgRPC
메시지 크기작은 JSON/REST 호출큰 payload, 바이너리 메시지
성능조금 느림 (텍스트 기반)빠름, 스트리밍 가능

async Task CallRvGatewayAsync(Protocol protocol, string serviceName, string methodName, object requestBody)
{

    {

        var reply = await daprClient.InvokeMethodGrpcAsync<object, string>(
            serviceName, methodName, requestBody);

        Console.WriteLine("gRPC Response: " + reply);
    }
    else
    {
        daprClient = new DaprClientBuilder()
            .UseHttpEndpoint("http://localhost:3500") // HTTP Dapr sidecar endpoint
            .Build();

        var reply = await daprClient.InvokeMethodAsync<object, string>(
            serviceName, methodName, requestBody, HttpMethod.Post);

        Console.WriteLine("HTTP Response: " + reply);
    }
}

  • protocol 매개변수로 호출 시 사용 프로토콜 선택 가능

  • Dapr Sidecar의 **HTTP 포트(기본 3500)**와 **gRPC 포트(기본 50001)**를 사용

2️⃣ 동적 선택 구현 패턴

  1. 환경 변수 기반

    • 예: USE_GRPC=true → gRPC 호출, 아니면 HTTP

  2. 런타임 설정 UI

    • 사용자 선택에 따라 HTTP/gRPC 전환

  3. 서비스 별 정책

    • 소규모 호출 → HTTP, 대용량/스트리밍 → gRPC

  • Dapr Service Invocation은 HTTP와 gRPC 모두 지원

  1. 대량 요청

    • 수천~수만 건 동시 호출 시 → gRPC 사용

  2. 실시간 데이터 스트리밍

    • 예: RV Gateway가 대량 이벤트를 받아 처리 → gRPC Server Streaming 또는 Bidirectional Streaming

  3. 작은 요청, 낮은 빈도

    • REST(HTTP)로 충분

    • 구현 간단, 기존 REST Controller 재사용 가능

1️⃣ HTTP/2와 Connection Reuse

  • HTTP/2는 단일 TCP 연결 위에서 멀티플렉싱을 지원합니다.

  • Dapr Service Invocation에서 gRPC(HTTP/2)를 사용하면, 클라이언트 사이드카 ↔ 서버 사이드카 간 연결은 한 번 생성 후 여러 요청이 같은 연결을 재사용합니다.

  • 즉, 클라이언트 사이드카 입장에서는 한 TCP 연결을 통해 여러 요청을 보내기 때문에 서버 파드가 여러 개 있어도, 기본 라운드로빈이 아닌 한 연결로 고정될 수 있음.

결과: 트래픽이 일부 서버 파드에 몰리는 것처럼 보일 수 있음.

2️⃣ 트래픽 분산 구조

요소동작
Dapr Sidecar (클라이언트)기본적으로 App ID 기준 라우팅. Service Invocation 시 서버 Pod 중 하나 선택
K8s ServiceClusterIP 로드밸런서가 라운드로빈 분산
gRPC (HTTP/2)Persistent Connection + 멀티플렉싱 → 동일 TCP 연결로 반복 요청 → 일부 Pod에 집중 가능

즉, gRPC Persistent Connection + HTTP/2 때문에 K8s Service 레벨에서 Pod 간 라운드로빈이 1회만 적용되고, 이후 요청은 같은 연결로 몰리는 현상이 발생할 수 있습니다.

3️⃣ 해결 방법

  1. Dapr Resiliency / Retry 설정

    • Dapr에서 maxRetries + timeout을 적절히 조정

    • 실패 시 다른 Pod로 요청 재시도 가능

  2. gRPC Channel 분리 / 커넥션 풀

    • Client App에서 여러 gRPC 채널 생성 → 여러 연결을 동시에 열어라

    • Dapr SDK에서는 기본적으로 1~2개의 채널을 쓰므로, 필요시 다수 채널 생성

  3. K8s Service 타입 활용

    • 기본 ClusterIP: Pod 선택시 Round-Robin

    • gRPC Persistent Connection 문제를 완전히 피하려면, External Load Balancer + Session Affinity 없음 사용

  4. Dapr Placement/Invocation Policy

    • Dapr 2.x 이상에서 invocation policy 설정 가능

    • 예: round-robin 또는 least-connection 기반 분산 정책

  5. Scaling + Horizontal Pods

    • Pod 수를 늘리고, 클라이언트 사이드카를 여러 인스턴스로 운영 → 트래픽 자동 분산

부하 불균형K8s ClusterIP 한 번만 라운드로빈External LB, session affinity off, HPA scaling
멀티 파드 환경에서 안정REST(HTTP/1.1) 사용단순 라운드로빈, Persistent Connection 없음

네, gRPC 자체에는 자동 재연결(Automatic Reconnect) 기능이 내장되어 있습니다. 다만, 몇 가지 조건과 주의사항이 있습니다. Dapr 환경에서 gRPC를 사용하는 경우를 기준으로 설명드릴게요.

1️⃣ gRPC Automatic Reconnect 기본 동작

  • gRPC 클라이언트는 TCP 연결이 끊기면 자동으로 서버에 재연결을 시도합니다.

  • 기본 재연결 전략(Backoff 정책)은 대부분 Exponential Backoff + Max Delay 형태입니다.

  • 연결이 끊기더라도 gRPC 스트림 또는 unary 호출이 재시도되는 것은 연결 재성립 이후부터 가능합니다.

  • Bidirectional Streaming의 경우 연결이 끊기면 스트림도 끊기므로, 클라이언트에서 다시 스트림을 생성해야 합니다.

2️⃣ Dapr 환경에서 gRPC 호출

  • Client App → Dapr SDK → Dapr Sidecar → Server Sidecar → App

  • Dapr Service Invocation을 gRPC로 호출할 경우:

    1. Client Sidecar ↔ Server Sidecar 연결: gRPC TCP 연결

    2. 연결이 끊기면 Dapr Sidecar가 자동으로 다른 Pod 또는 같은 Pod로 재연결

    3. Client SDK에서 호출 재시도 정책이 적용되어야 완전히 투명하게 처리됨

즉, Dapr 자체는 서비스 발견 + 로드밸런싱을 수행하고, gRPC는 TCP 연결 재수립을 담당합니다.

3️⃣ 주의 사항

항목주의
Unary 호출자동 재연결 후 다음 호출부터 정상 처리 가능
Streaming 호출기존 스트림 끊기면 새 스트림 생성 필요 (클라이언트 로직 처리)
Sidecar 재시도Dapr SDK에서는 InvokeMethodGrpcAsync 호출 시 retry/cancellation 설정 필요
Load Balancing한 연결이 특정 Pod로 고정되므로, 재연결 시 다른 Pod로 연결될 수 있음
연결 끊김 감지gRPC는 Status 코드 UNAVAILABLE를 반환, 클라이언트에서 재시도 로직 필요

4️⃣ 권장 패턴

  1. Unary 요청: gRPC 기본 재연결 + Dapr Resiliency Policy(retry, timeout) → 대부분 문제 없음

  2. Streaming 요청:

    • 끊김 감지 (OnStatusChanged)

    • 스트림 재생성 + 재구독 로직 필요

  3. 멀티 Pod 환경:

    • Dapr Service Invocation Policy 설정 → 연결 재수립 시 Pod 균등 분산 가능

  4. 대량 처리:

    • 연결을 여러 개 만들어 멀티플렉싱 → 한 연결 끊겨도 일부 요청 처리 지속

✅ 결론:

  • gRPC는 기본적으로 자동 재연결 지원

  • Unary 호출은 거의 투명하게 처리 가능

  • Streaming/실시간 데이터는 끊김 감지 + 재구독 로직 필요

  • Dapr과 함께 쓰면 서비스 재발견과 Pod 로드밸런싱까지 자동 지원

enum Protocol { HTTP, GRPC }

async Task CallRvGatewayAsync(Protocol protocol, string serviceName, string methodName, object requestBody)
{
    DaprClient daprClient;

    if (protocol == Protocol.GRPC)
    {
        daprClient = new DaprClientBuilder()
            .UseGrpcEndpoint("http://localhost:50001")
